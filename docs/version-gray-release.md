# 版本管理与灰度发布 — 需求文档

## 一、功能概述

### 1.1 版本管理
为 AI 在线服务提供多版本生命周期管理能力。每个服务可同时存在多个版本（如 v1、v2、v3），支持版本的创建、发布、回滚、下线等操作，确保模型迭代过程中服务持续可用。

### 1.2 灰度发布
在服务新版本上线时，通过流量分配策略将部分请求路由到新版本，逐步验证稳定性后再全量切换，降低发布风险。支持按流量比例、按特定用户标签等灰度策略。

## 二、版本管理

### 2.1 核心概念

| 概念 | 说明 |
|------|------|
| 服务 | 顶层实体，一个服务对应一个 AI 模型能力（如"文本生成"、"图像识别"） |
| 服务版本 | 服务的具体版本实例，包含模型文件、配置、部署信息。同一服务下可存在多个版本 |
| 当前版本 | 对外提供服务的正式版本，仅可有一个 |
| 历史版本 | 曾经作为当前版本、现已下线的版本，用于回滚参考 |
| 草稿版本 | 正在编辑配置但尚未发布的版本 |

### 2.2 版本状态流转

```
草稿(draft) → 发布中(releasing) → 运行中(running)  ← 当前版本
                                   ↓
                              已停止(stopped)
                                   ↓
                              已下线(offline)   → 历史版本
```

- **草稿**：创建新版本后的初始状态，可编辑配置
- **发布中**：正在部署模型和启动实例
- **运行中**：正常对外提供服务
- **已停止**：手动停止或异常停止
- **已下线**：被新版本替代后下线

### 2.3 数据字段

#### 服务版本表（service_version）

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| version_id | string | 是 | 版本唯一标识，格式 `svc_{service_id}_v{n}` |
| service_id | string | 是 | 所属服务ID |
| version_name | string | 是 | 版本名称，如 v1、v2，用户可自定义 |
| version_number | int | 是 | 版本序号，自增，不可修改 |
| model_name | string | 是 | 使用的模型名称 |
| model_version | string | 是 | 模型版本标识 |
| adapter_count | int | 是 | 适配器数量 |
| instance_count | int | 是 | 实例数量 |
| status | enum | 是 | 版本状态：draft / releasing / running / stopped / offline |
| is_current | boolean | 是 | 是否为当前正式版本 |
| deploy_method | enum | 是 | 部署方式：online_deploy / managed_import |
| config_snapshot | json | 否 | 版本配置快照（实例规格、自动扩缩容策略、环境变量等） |
| gray_status | enum | 否 | 灰度状态：none / graying / gray_completed |
| gray_config | json | 否 | 灰度发布配置（详见灰度发布章节） |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 最后更新时间 |
| published_at | datetime | 否 | 发布时间 |
| offline_at | datetime | 否 | 下线时间 |
| created_by | string | 是 | 创建人 |
| description | string | 否 | 版本说明/变更日志 |

#### 版本配置快照（config_snapshot）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| instance_spec | string | 实例规格，如 GPU型号、显存 |
| min_instances | int | 最小实例数 |
| max_instances | int | 最大实例数 |
| scale_policy | string | 扩缩容策略：manual / auto |
| env_vars | object | 环境变量键值对 |
| timeout | int | 请求超时时间（秒） |
| max_concurrency | int | 最大并发数 |

### 2.4 交互说明

#### 2.4.1 版本列表入口
- 在服务管理列表中，点击某服务行的"详情"进入服务详情页
- 服务详情页顶部为 Tab 导航，包含"版本管理"Tab
- 点击"版本管理"Tab 进入版本列表

#### 2.4.2 版本列表页
- **布局**：表格形式，展示当前服务下所有版本
- **列定义**：版本名称、模型/版本号、适配器数量、实例数量、版本状态、是否当前版本、灰度状态、创建时间、操作
- **当前版本标识**：在"版本名称"列显示"当前版本"标签（绿色背景）
- **排序**：默认按版本序号降序（最新版本在前）
- **筛选**：支持按版本状态筛选
- **操作按钮**（按状态区分）：
  - 草稿：编辑、发布、删除
  - 运行中：停止、配置灰度（非当前版本时）、下线
  - 已停止：启动、下线
  - 已下线：重新发布

#### 2.4.3 创建新版本
- 点击"新建版本"按钮，弹出创建表单
- **表单字段**：
  - 版本名称（文本输入，默认自动生成 v{n+1}）
  - 版本说明（多行文本）
  - 模型选择（下拉选择，复用当前服务绑定的模型列表）
  - 部署方式（下拉选择）
  - 实例规格（下拉选择）
  - 初始实例数（数字输入）
- 提交后版本状态为"草稿"

#### 2.4.4 发布版本
- 在草稿版本上点击"发布"
- 弹出确认对话框，提示：
  - "发布后该版本将启动实例并对外提供服务"
  - 如果当前已有运行中的版本，提示"是否同时配置灰度发布？是 → 进入灰度配置；否 → 直接全量替换"
- 确认后版本状态变为"发布中"，后台开始部署
- 部署完成后状态变为"运行中"

#### 2.4.5 版本回滚
- 在已下线的历史版本上点击"回滚"
- 弹出确认框："回滚到 {version_name}，当前版本将被下线，确认回滚？"
- 确认后：当前版本下线，目标版本重新启动

#### 2.4.6 版本对比
- 在版本列表中选择两个版本，点击"对比"按钮
- 侧滑面板展示两版本配置差异（左右对照），差异项高亮标记
- 可对比内容：模型版本、实例规格、扩缩容策略、环境变量、超时/并发配置

## 三、灰度发布

### 3.1 核心概念

| 概念 | 说明 |
|------|------|
| 灰度发布 | 将部分流量按策略路由到新版本进行验证的发布方式 |
| 基线版本 | 当前正式对外提供服务的版本（全量流量） |
| 灰度版本 | 正在验证中的新版本，接收灰度流量 |
| 灰度规则 | 定义流量分配策略，决定哪些请求路由到灰度版本 |
| 全量发布 | 验证通过后，将灰度版本设为新的基线版本，接收 100% 流量 |

### 3.2 灰度状态流转

```
未灰度(none) → 灰度中(graying) → 灰度完成(gray_completed)
                     ↓                    ↓
                  取消灰度            全量发布
                (回退到 none)      (新版本成为当前版本)
```

### 3.3 数据字段

#### 灰度配置（gray_config）

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| gray_id | string | 是 | 灰度任务唯一标识 |
| service_id | string | 是 | 服务ID |
| baseline_version_id | string | 是 | 基线版本ID |
| gray_version_id | string | 是 | 灰度版本ID |
| strategy_type | enum | 是 | 灰度策略类型：traffic_ratio / user_tag / header_match |
| strategy_config | json | 是 | 策略详细配置（见下） |
| status | enum | 是 | 灰度状态：graying / completed / cancelled |
| started_at | datetime | 是 | 灰度开始时间 |
| completed_at | datetime | 否 | 灰度完成/取消时间 |
| created_by | string | 是 | 发起人 |

#### 灰度策略配置（strategy_config）

**策略一：按流量比例（traffic_ratio）**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| gray_ratio | int | 灰度流量占比（1-100），如 10 表示 10% 流量到灰度版本 |
| step_config | object | 分步放量配置（可选） |

**分步放量配置（step_config）**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| enabled | boolean | 是否启用分步放量 |
| steps | array | 放量步骤列表 |
| steps[].ratio | int | 该步骤的目标流量比例 |
| steps[].duration_minutes | int | 该步骤持续时间（分钟），到期后自动进入下一步骤 |
| steps[].auto_proceed | boolean | 是否自动进入下一步，false 时需手动确认 |

**策略二：按用户标签（user_tag）**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| tag_key | string | 用户标签键名，如 "user_level" |
| tag_values | array | 匹配的标签值列表，如 ["vip", "beta_tester"] |
| match_mode | enum | 匹配模式：include（包含即命中）/ exact（精确匹配） |

**策略三：按请求头匹配（header_match）**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| header_key | string | 请求头键名，如 "X-Gray-Tag" |
| header_values | array | 匹配的请求头值列表 |
| match_mode | enum | 匹配模式：include / exact / regex |

### 3.4 交互说明

#### 3.4.1 发起灰度发布

**入口1：版本列表操作**
- 在版本列表中，运行中（非当前版本）的版本行点击"配置灰度"
- 弹出灰度配置弹窗

**入口2：发布版本时选择**
- 发布新版本时，确认对话框中选择"灰度发布"（而非"全量替换"）
- 自动跳转到灰度配置弹窗

#### 3.4.2 灰度配置弹窗

- **基线版本**：只读展示，显示当前正式版本名称
- **灰度版本**：只读展示，显示待灰度的新版本名称
- **灰度策略选择**：Tab 切换三种策略
  - **流量比例**：滑块 + 数字输入，范围 1-100%，默认 10%
    - 可选"分步放量"：勾选后展示步骤列表，支持添加/删除步骤
    - 每步配置：目标比例、持续时间、是否自动进入下一步
  - **用户标签**：
    - 标签键名输入框
    - 标签值列表（支持多值输入，Tag 形式展示，可逐个删除）
    - 匹配模式下拉：包含 / 精确匹配
  - **请求头匹配**：
    - 请求头键名输入框
    - 值列表输入（同上）
    - 匹配模式下拉：包含 / 精确匹配 / 正则
- **高级设置**（折叠面板，默认收起）：
  - 灰度观察时长：设置自动提醒时间（如 30 分钟后提醒确认全量发布）
  - 异常熔断：开关 + 阈值配置（错误率超过 X% 自动回退到基线版本）
- **底部按钮**：取消、确认开始灰度

#### 3.4.3 灰度进行中状态

- 版本列表中，灰度版本行的灰度状态列显示"灰度中"标签（橙色）
- 点击"查看灰度详情"进入灰度监控面板

#### 3.4.4 灰度监控面板

- **顶部信息栏**：基线版本、灰度版本、当前策略、灰度开始时间、已持续时间
- **流量分配可视化**：
  - 双端条形图：左侧为基线版本流量占比，右侧为灰度版本流量占比
  - 实时更新当前各版本的请求 QPS
- **指标对比区域**（左右对照基线 vs 灰度）：
  - 请求成功率
  - 平均响应延迟（P50/P95/P99）
  - 错误率
  - GPU 利用率
  - 请求总数
- **操作按钮**：
  - "调整流量比例"：弹出滑块修改灰度流量占比（仅流量比例策略）
  - "全量发布"：确认后将灰度版本提升为当前版本，基线版本下线
  - "取消灰度"：确认后灰度版本停止，所有流量回归基线版本

#### 3.4.5 全量发布确认

- 点击"全量发布"弹出确认对话框：
  - "确认将 {灰度版本名} 设为当前版本？"
  - "基线版本 {基线版本名} 将被下线"
  - 二次确认输入框：输入版本名确认
- 确认后：灰度版本成为当前版本（is_current = true），基线版本下线

#### 3.4.6 取消灰度

- 点击"取消灰度"弹出确认对话框：
  - "取消灰度后，所有流量将回归基线版本 {基线版本名}，灰度版本 {灰度版本名} 将停止运行"
- 确认后：灰度版本停止，流量 100% 路由到基线版本

#### 3.4.7 灰度历史记录

- 版本管理页面的"灰度历史"Tab
- 表格展示历史灰度记录：灰度ID、基线版本、灰度版本、策略类型、开始时间、结束时间、结果（全量发布 / 取消灰度）、操作人
- 点击行可展开查看灰度期间的指标数据

## 四、整体交互流程图

```
创建新版本(草稿) → 编辑配置 → 发布
                              ↓
                    ┌── 全量替换（直接替换当前版本）
                    │
                    └── 灰度发布
                           ↓
                     配置灰度策略（流量比例/用户标签/请求头）
                           ↓
                     灰度进行中 → 监控指标
                           ↓
                  ┌────────┴────────┐
              全量发布            取消灰度
                  ↓                  ↓
          新版本成为当前版本    流量回归基线版本
          旧版本下线          灰度版本停止
```

## 五、接口设计（参考）

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 版本列表 | GET | /api/services/{serviceId}/versions | 获取服务下所有版本 |
| 创建版本 | POST | /api/services/{serviceId}/versions | 创建新版本（草稿） |
| 版本详情 | GET | /api/services/{serviceId}/versions/{versionId} | 获取版本详情及配置 |
| 更新版本配置 | PUT | /api/services/{serviceId}/versions/{versionId}/config | 更新草稿版本配置 |
| 发布版本 | POST | /api/services/{serviceId}/versions/{versionId}/publish | 发布版本 |
| 停止版本 | POST | /api/services/{serviceId}/versions/{versionId}/stop | 停止运行中的版本 |
| 下线版本 | POST | /api/services/{serviceId}/versions/{versionId}/offline | 下线版本 |
| 回滚版本 | POST | /api/services/{serviceId}/versions/{versionId}/rollback | 回滚到指定版本 |
| 版本对比 | GET | /api/services/{serviceId}/versions/compare?v1={id1}&v2={id2} | 对比两个版本配置差异 |
| 发起灰度 | POST | /api/services/{serviceId}/gray | 发起灰度发布 |
| 灰度详情 | GET | /api/services/{serviceId}/gray/current | 获取当前灰度任务详情 |
| 调整灰度流量 | PUT | /api/services/{serviceId}/gray/traffic | 调整灰度流量比例 |
| 全量发布 | POST | /api/services/{serviceId}/gray/promote | 灰度转全量发布 |
| 取消灰度 | POST | /api/services/{serviceId}/gray/cancel | 取消灰度发布 |
| 灰度历史 | GET | /api/services/{serviceId}/gray/history | 灰度发布历史记录 |
